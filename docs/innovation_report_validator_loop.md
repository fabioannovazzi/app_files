# Innovation Report Validator Loop

This document is the fixed operating loop for the innovation monitoring report
validator. Do not change the loop while doing validator work unless the change is
itself the explicit task.

## Scope

The validator is for innovation monitoring report decks. It is not for arbitrary
presentation decks.

The source package is deterministic. The report deck text is generated by an LLM
from that package. The validator checks the LLM-generated report text against
the deterministic package and makes the reasoning visible.

Other report types can have their own packages and checks.

## Test Loop

1. Start from the innovation monitoring report decks we have.

2. Check that the deck reading worked:
   - OCR exists.
   - LLM slide mapping exists.
   - OCR text is preserved in the LLM mapping.
   - If this fails, do not trust the deck as input yet.

3. Use the mapped text units as the working input.

4. Go through all text units in the decks.

5. Run the existing deterministic filters/checkers on those text units.

6. Build a Pro audit package. The package contains the prompt separately from
   the data. It also contains a data-only zip without the prompt, so Pro can
   receive the structured artifacts cleanly.

7. Run Pro prompt 1: audit the current deterministic validator behavior. Pro
   audits the current deterministic result and defines the expected behavior for
   the test run:
   - it is a real innovation-report claim or a false positive
   - the rule/claim family is correct or wrong
   - the rule is generic enough for this report type
   - the cited deterministic package source is plausible or suspicious
   - expected value, observed value, denominator, tolerance, and reason are visible
   - remaining uncaught/unresolved units are non-claim, OCR/layout/mapping
     problem, or real residual claim

8. Pro prompt 1 also classifies remaining uncaught and unresolved units as:
   - non-claim
   - OCR/layout/mapping problem
   - real residual claim

9. Run Pro prompt 2: convert Pro prompt 1 output into an implementation brief.
   Prompt 2 receives the same data package plus Pro prompt 1 JSON output. It
   groups the audit findings into implementation work:
   - deterministic filters/checkers to add or fix
   - source mappings to repair
   - false-positive filters to tighten
   - evidence fields that must be exposed
   - items that should remain unresolved
   - OCR/layout/mapping issues to investigate
   - test expectations for the Python implementation

10. Implement from the structured audit and implementation brief. Pro defines
   the test expectation; it does not write the production validator.
   Implementation can:
   - fix false-positive filters
   - fix wrong rule routing or source mappings
   - add missing evidence fields
   - define reusable filters for repeated residual claims
   - write Python checkers for residual claims that can be computed from package
     data

11. For repeated residual claims, decide if they can become deterministic:
   - define a reusable filter
   - define the Python checker
   - tie it to package data

12. If a residual claim cannot yet be computed deterministically, use the LLM
    only to interpret/classify it in a structured way, not as final validation
    truth.

13. Re-run the deterministic validator on the same decks and compare the new
    output against Pro-defined expectations:
    - expected caught claims are caught
    - false positives are gone
    - wrong mappings are fixed
    - residual/non-claim classifications are respected
    - evidence is visible
    - remaining unresolved items are intentional

14. Repeat on another report set to test whether the rules generalize within
    innovation monitoring reports.

15. Show the result on top of or alongside the report text, so the system is not
    black box:
    - approved deterministic claims
    - failed deterministic claims
    - residual claims
    - non-claim text
    - OCR/layout/mapping issues

16. Only after this discuss whether a deck is good or bad.

## Pro Prompt Roles

Pro is part of the test loop only.

Prompt 1 is the audit prompt. It uses the data-only zip and answers whether the
current deterministic validator behaved correctly on caught, unresolved,
uncaught, and unmatched items. Its output is structured JSON and acts as a test
oracle.

Prompt 2 is the implementation-brief prompt. It uses the same data-only zip plus
Prompt 1 JSON output. It does not write code. It groups the audit findings into
deterministic implementation work and test expectations.

Neither prompt belongs in production runtime. Production remains deterministic.

## Production Validator

The production validator is not this loop.

At runtime, the validator should:

1. Load the mapped deck text.
2. Check reading completeness.
3. Run deterministic filters/checkers.
4. Return claim results, unresolved items, reading quality, and evidence.
5. Expose results in an auditable UI/overlay.

At runtime, the validator should not ask Pro what to do, learn new rules, or
change expectations dynamically.

## Visual Inspection Surface

The final result should be inspectable on the report itself, using an overlay or
side-by-side surface similar to the existing slide editor OCR overlay.

The surface should show the mapped report text and the validator decision for
each unit, including deterministic claims, failed deterministic claims, residual
claims, non-claim text, and OCR/layout/mapping issues.

## LLM Use

LLM output can help interpret or classify residual text in the test loop. It is
not deterministic validation truth.

Once a repeated wording pattern can be computed from package data, encode it as
a deterministic filter and Python checker.
